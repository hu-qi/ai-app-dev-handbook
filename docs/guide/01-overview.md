# 01. AI 应用开发全景图

AI 应用不是在传统系统里加一个聊天框，而是把模型能力、业务数据、外部工具和产品体验组合成一个可控系统。

## 一、AI 应用的五层架构

### 1. 交互层

交互层决定用户如何向 AI 表达任务，也决定 AI 如何返回结果。常见形态包括聊天窗口、表单式任务、侧边栏助手、编辑器插件、知识库问答框和自动化工作台。

设计交互层时，不要只问“能不能聊天”，而要问：

- 用户输入是否足够结构化？
- 是否需要上传文件？
- 是否需要实时流式输出？
- 是否需要展示引用来源？
- 是否需要人工确认后再执行动作？

### 2. 编排层

编排层负责把用户任务拆成可执行步骤。简单场景可以直接调用模型，复杂场景需要工作流、工具调用、任务队列或 Agent。

典型编排方式：

- 单次模型调用：适合摘要、改写、分类。
- 多步骤链路：适合先检索、再分析、再生成。
- 工作流：适合规则明确、流程固定的企业场景。
- Agent：适合任务开放、需要动态决策的场景。

### 3. 模型层

模型层负责理解、推理、生成和调用工具。工程上不要把模型写死，要设计模型网关，方便切换不同供应商。

模型选择需要看：

- 中文能力
- 长上下文能力
- 结构化输出稳定性
- 工具调用能力
- 响应速度
- 调用成本
- 私有化或合规要求

### 4. 数据层

数据层是 AI 应用能否落地的关键。企业 AI 应用通常不是缺模型，而是缺高质量、可检索、可追溯、可权限控制的数据。

数据层包括：

- 原始文档
- 结构化业务数据
- 向量索引
- 用户画像
- 对话历史
- 工具执行结果
- 评测数据集

### 5. 运维层

AI 应用上线后必须可观测。你需要知道每一次调用的输入、输出、模型、耗时、Token、费用、错误、用户反馈和版本。

否则你无法判断：

- 为什么回答错了？
- 是模型问题、检索问题还是 Prompt 问题？
- 成本为什么突然升高？
- 哪些用户问题最值得产品化？

## 二、AI 应用的三种主流形态

### 1. Copilot 型

特点是辅助用户完成任务，不完全替代用户决策。例如代码助手、写作助手、运营助手、数据分析助手。

适合场景：

- 用户有明确工作流
- AI 作为效率增强工具
- 结果需要人工确认

### 2. Knowledge Base 型

特点是围绕企业文档、知识库和业务资料进行问答、总结、检索和溯源。

适合场景：

- 内部制度问答
- 产品资料问答
- 客服知识库
- 技术文档助手

### 3. Agent 型

特点是 AI 能够根据目标自主拆解任务、调用工具并迭代结果。

适合场景：

- 研究分析
- 数据收集
- 多系统操作
- 自动化报告

但要注意：Agent 不是越自由越好。真正可落地的 Agent 通常需要明确边界、工具白名单、状态机和人工确认点。

## 三、从 Demo 到产品的关键差异

Demo 只需要“看起来能用”，产品需要“长期稳定可用”。

| 维度 | Demo | 产品 |
| --- | --- | --- |
| 模型 | 单模型硬编码 | 模型网关与降级 |
| Prompt | 写在代码里 | 版本管理与评测 |
| 数据 | 少量测试文档 | 权限、同步、清洗、索引 |
| 调用 | 手动触发 | 队列、重试、限流 |
| 结果 | 只看效果 | 可追溯、可反馈、可回滚 |
| 成本 | 不敏感 | Token、缓存、预算控制 |

## 四、一个最小 AI 应用应该包含什么

最小可用版本建议包含：

1. 用户输入界面
2. 后端模型调用接口
3. Prompt 模板
4. 流式输出
5. 错误提示
6. 调用日志
7. 基础权限
8. 成本记录

不要一开始就上复杂 Agent。先把最小闭环跑通，再逐步加入知识库、工具调用和工作流。

## 五、本章实践任务

请画出你想做的 AI 应用架构图，至少包含：

- 用户入口
- 后端服务
- 模型供应商
- 数据来源
- 工具系统
- 日志与评测

完成后，回答三个问题：

1. 这个应用解决谁的什么问题？
2. 模型在哪个环节产生核心价值？
3. 如果模型答错，系统如何发现和修正？
