# 07. 模型评测与可观测性

AI 应用上线后，不能只靠用户反馈判断好坏。你需要建立评测与可观测体系，持续回答三个问题：效果怎么样、为什么失败、怎么改进。

## 一、为什么评测重要

AI 应用的输出具有不确定性。同一个问题，在不同模型、Prompt、上下文和温度下都可能得到不同答案。

没有评测，你只能凭感觉调 Prompt；有了评测，你才能像优化工程系统一样优化 AI 系统。

## 二、评测对象

需要评测的不只是最终答案，还包括：

- 意图识别是否正确
- 检索是否命中资料
- 工具参数是否正确
- 工作流节点是否按预期执行
- 最终答案是否准确
- 引用来源是否可靠
- 输出格式是否稳定

## 三、评测集设计

一个基础评测集应包含：

1. 典型问题
2. 边界问题
3. 恶意输入
4. 信息不足问题
5. 多轮上下文问题
6. 真实用户失败案例

每条样本应包含：输入、期望答案、评分标准、相关资料和标签。

## 四、自动评测与人工评测

自动评测适合格式、关键词、引用、工具参数等可规则化指标。人工评测适合判断准确性、表达质量和业务可接受度。

更成熟的做法是 LLM-as-Judge，但要注意评审模型也会犯错，因此重要场景仍需人工抽检。

## 五、可观测日志

每次调用建议记录：

- requestId
- userId
- 场景
- 模型
- Prompt 版本
- 输入 Token
- 输出 Token
- 总费用
- 延迟
- 检索命中的 Chunk
- 工具调用记录
- 最终输出
- 用户反馈

## 六、失败归因

AI 应用失败通常不是单点问题。常见归因：

- 用户问题不清晰
- Prompt 约束不足
- 检索未命中
- 上下文过长
- 模型能力不足
- 工具返回错误
- 输出格式不稳定

## 七、迭代闭环

推荐流程：

1. 收集失败案例
2. 标注失败原因
3. 修改 Prompt、检索或工具逻辑
4. 跑回归评测
5. 小流量上线
6. 观察线上指标

## 八、本章实践任务

为你的知识库问答系统建立一个 30 条样本的评测集，并记录：

- 问题
- 标准答案
- 相关文档
- 模型答案
- 评分
- 失败原因
- 改进建议
