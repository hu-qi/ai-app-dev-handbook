# 02. 大模型 API 调用基础

大模型 API 是 AI 应用的发动机。你需要掌握的不只是如何发起一次请求，还要理解消息结构、上下文管理、流式输出、结构化结果、异常重试和成本控制。

## 一、核心调用模型

一次典型调用包含：

- system：定义角色、边界和规则
- user：用户输入
- assistant：历史回复
- tool：工具执行结果

在真实业务中，system prompt 不应该被用户覆盖，用户输入也不应该直接拼进高权限指令里。

## 二、消息结构设计

推荐把 Prompt 拆成四层：

1. 角色：你是谁
2. 任务：要完成什么
3. 约束：不能做什么，输出格式是什么
4. 上下文：业务数据、文档片段、用户信息

示例：

```ts
const messages = [
  { role: 'system', content: '你是企业知识库问答助手，只能基于给定资料回答。' },
  { role: 'user', content: question }
]
```

## 三、流式输出

流式输出能显著改善用户体验。对于长答案、代码生成、报告生成，用户不应该等待所有内容完成后才看到结果。

工程实现通常使用：

- Server-Sent Events
- WebSocket
- Fetch ReadableStream

设计流式输出时要考虑：

- 中途失败如何提示
- 是否保存完整输出
- 前端如何处理 Markdown
- 如何统计最终 Token

## 四、结构化输出

AI 应用经常需要模型返回 JSON，例如分类、标签、表单字段、工具参数。此时不能只靠“请返回 JSON”，还要做校验和重试。

建议流程：

1. 定义 JSON Schema
2. 要求模型按 Schema 输出
3. 服务端解析 JSON
4. 校验字段类型
5. 失败时带错误原因重试

## 五、上下文管理

上下文不是越长越好。上下文越长，成本越高，干扰越多，延迟越大。

常见策略：

- 最近 N 轮对话保留
- 长对话摘要压缩
- 关键事实写入 Memory
- 与当前问题无关的信息不注入
- RAG 场景只注入高相关片段

## 六、错误处理

必须处理这些错误：

- 网络超时
- 供应商限流
- 模型返回空内容
- JSON 解析失败
- 工具参数不合法
- 内容安全拦截
- 余额不足或 Key 失效

推荐策略：

- 短暂重试
- 指数退避
- 模型降级
- 错误日志
- 用户友好提示

## 七、成本控制

成本来自输入 Token、输出 Token、Embedding、Rerank、工具系统和存储。

优化方式：

- Prompt 精简
- 上下文裁剪
- 结果缓存
- 小模型处理简单任务
- 大模型只处理复杂推理
- 对长任务设置预算上限

## 八、本章实践任务

实现一个最小聊天接口，要求：

- 支持 system prompt
- 支持流式输出
- 记录输入、输出、耗时
- 捕获异常并返回友好错误
- 为每次调用生成 requestId
