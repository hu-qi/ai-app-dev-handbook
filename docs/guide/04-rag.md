# 04. RAG 知识库架构

RAG 是 Retrieval-Augmented Generation 的缩写，即检索增强生成。它的核心思想是：先从外部知识库中检索相关资料，再把资料作为上下文交给大模型回答。

## 一、为什么需要 RAG

大模型本身存在三个问题：

- 不知道企业私有知识
- 训练数据可能过时
- 容易编造看似合理的答案

RAG 的价值是把回答约束在可追溯资料之内，让模型从“凭记忆回答”变成“基于证据回答”。

## 二、RAG 的基本流程

1. 文档采集
2. 文档解析
3. 文本切分
4. 向量化
5. 存入向量数据库
6. 用户提问
7. 检索相关片段
8. 组装 Prompt
9. 模型生成答案
10. 返回答案和引用来源

## 三、文档解析

不同文档需要不同解析方式：

- Markdown：保留标题层级
- PDF：注意页码、表格和图片
- Word：注意段落、标题、表格
- HTML：清理导航、广告和脚本
- Excel：按 sheet、行列和字段解释

解析的目标不是“把字提取出来”，而是保留结构。

## 四、文本切分

Chunk 切分直接影响召回质量。

常见策略：

- 按固定长度切分
- 按标题层级切分
- 按语义段落切分
- 滑动窗口重叠
- 父子 Chunk

建议：企业文档优先按标题层级切分，再对长段落做二次切分。

## 五、Embedding 与向量库

Embedding 把文本转换成向量，向量库负责相似度检索。

常见选择：

- pgvector：适合已有 PostgreSQL 团队
- Qdrant：部署简单，性能不错
- Milvus：适合大规模向量检索
- Elasticsearch：适合混合检索

## 六、检索优化

基础向量检索不一定够用。常见优化：

- Query Rewrite：改写用户问题
- Hybrid Search：关键词 + 向量
- Rerank：重排候选片段
- Metadata Filter：按权限、时间、类别过滤
- Context Compression：压缩上下文

## 七、回答约束

RAG 的 Prompt 必须强调：

- 只能基于给定资料回答
- 无资料时要说明不知道
- 必须引用来源
- 不要编造文件名、页码和数据

## 八、评测指标

RAG 评测至少看：

- 召回是否命中正确资料
- 答案是否基于资料
- 是否遗漏关键点
- 是否产生幻觉
- 引用是否准确
- 响应速度是否可接受

## 九、本章实践任务

构建一个最小企业知识库：

- 支持上传 Markdown 文档
- 自动切分并向量化
- 用户提问后检索 Top 5 片段
- 生成带引用来源的回答
- 记录每次命中的 Chunk
