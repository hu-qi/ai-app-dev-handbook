# 06. Agent 工作流

Agent 是能够围绕目标进行任务拆解、工具调用、结果检查和迭代优化的 AI 系统。但在真实业务中，Agent 不是越自由越好，而是越可控越有价值。

## 一、Agent 与普通聊天的区别

普通聊天是用户问、模型答。Agent 则需要：

- 理解目标
- 拆解任务
- 选择工具
- 执行动作
- 观察结果
- 判断是否继续
- 输出最终结果

## 二、Agent 的核心组件

### 1. Planner

负责把目标拆成步骤。

### 2. Executor

负责执行具体动作，例如调用搜索、数据库、文件系统或业务接口。

### 3. Memory

负责保存中间状态、用户偏好、任务记录和长期知识。

### 4. Critic

负责检查结果是否满足目标，避免错误继续扩大。

### 5. Controller

负责限制循环次数、工具权限、预算和停止条件。

## 三、工作流优先原则

很多业务不需要 Agent，只需要工作流。

如果流程明确、规则稳定、节点可枚举，优先使用工作流。如果任务开放、信息不完整、需要动态决策，再考虑 Agent。

## 四、常见 Agent 架构

### ReAct

模型不断进行 Thought、Action、Observation 循环。适合探索型任务，但需要限制循环次数。

### Plan and Execute

先生成计划，再逐步执行。适合复杂任务，但计划可能需要动态调整。

### Multi-Agent

多个 Agent 分工协作，例如研究员、分析师、审稿人。适合复杂内容生产和研究任务。

## 五、Agent 的风险

- 无限循环
- 工具误用
- 成本失控
- 编造中间结果
- 对错误观察继续推理
- 权限过大
- 输出不可复现

## 六、落地建议

1. 限制最大步骤数
2. 每次工具调用都记录日志
3. 高风险动作必须人工确认
4. 任务状态必须可视化
5. 失败时能恢复或回滚
6. 先用小模型做分类和路由，大模型做复杂推理

## 七、本章实践任务

实现一个“资料研究助手”：

- 用户输入研究主题
- Planner 拆解 3 到 5 个子问题
- Executor 分别收集资料摘要
- Critic 检查是否覆盖主题
- 最终生成结构化研究报告
