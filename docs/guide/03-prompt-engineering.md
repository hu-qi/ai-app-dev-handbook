# 03. Prompt 工程实战

Prompt 工程不是写一句神奇咒语，而是把任务、边界、上下文和输出格式工程化。

## 一、Prompt 的基本结构

一个稳定 Prompt 通常包含：

1. 角色定位
2. 任务目标
3. 输入说明
4. 输出格式
5. 约束条件
6. 示例
7. 失败处理规则

模板：

```text
你是一个{角色}。
你的任务是：{任务}。
请基于以下输入完成任务：{输入}。
输出必须满足：{格式}。
不要：{限制}。
如果信息不足，请说明缺失信息。
```

## 二、角色不是装饰

角色的价值是压缩上下文，让模型知道应该采用什么标准判断结果。

例如“你是资深 HR”不够明确，更好的写法是：

```text
你是熟悉跨境电商组织管理的人力 BP，擅长把业务目标拆成可执行的绩效指标。
```

## 三、输出格式必须可解析

如果结果要被程序消费，优先使用 JSON 或 Markdown 表格。

但注意：Markdown 适合人读，JSON 适合机器读。

## 四、Few-shot 示例

示例能显著提升稳定性。示例应该覆盖正常情况、边界情况和错误情况。

示例不是越多越好，关键是代表性强。

## 五、Prompt 版本管理

Prompt 应该像代码一样管理版本。

建议记录：

- prompt_id
- version
- 适用场景
- 输入变量
- 输出格式
- 修改原因
- 评测结果

## 六、Prompt 评测

不要凭感觉判断 Prompt 好坏。至少准备 20 条真实样本，人工标注期望结果，然后比较不同版本表现。

评测维度：

- 正确性
- 完整性
- 一致性
- 可执行性
- 格式稳定性
- 幻觉率

## 七、常见失败模式

### 1. 任务过大

把“帮我做一个商业计划书”拆成：市场分析、用户画像、产品定位、功能清单、商业模式、风险评估。

### 2. 约束冲突

同时要求“非常详细”和“控制在 100 字以内”，模型会摇摆。

### 3. 上下文污染

把无关资料塞进 Prompt，会让模型错误引用。

### 4. 输出不可控

没有格式约束，后端就很难解析。

## 八、本章实践任务

为一个“AI 简历优化助手”设计三版 Prompt：

1. 面向应届生
2. 面向 3 年经验开发者
3. 面向管理岗

每版都要包含：角色、任务、输入、输出格式、限制条件和示例。
