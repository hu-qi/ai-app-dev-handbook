# 05. Function Calling 与工具调用

工具调用让模型不只会说话，还能连接外部系统。它是 AI 应用从“内容生成”走向“任务执行”的关键能力。

## 一、工具调用解决什么问题

模型本身不能直接查数据库、发邮件、创建订单、读取实时天气或调用内部系统。工具调用的作用是：

1. 让模型判断是否需要工具
2. 生成符合 Schema 的工具参数
3. 由后端执行真实工具
4. 把执行结果交给模型继续推理

## 二、工具定义

一个工具至少包含：

- name：工具名称
- description：工具用途
- parameters：参数结构
- handler：真实执行逻辑
- permission：权限边界
- timeout：超时限制

工具描述必须清晰，否则模型会误用。

## 三、参数 Schema

参数 Schema 是工具调用稳定性的核心。字段类型、必填项、枚举值和描述都要明确。

不要让模型自由生成 SQL、脚本或未受控命令。工具参数必须经过服务端校验。

## 四、执行闭环

推荐流程：

1. 用户提出任务
2. 模型判断需要调用工具
3. 模型生成工具参数
4. 服务端校验参数
5. 服务端执行工具
6. 工具返回结构化结果
7. 模型基于结果生成回复
8. 记录工具调用日志

## 五、工具权限

工具越强，风险越高。

高风险工具包括：

- 支付
- 删除数据
- 发送外部消息
- 修改权限
- 执行代码
- 访问敏感信息

这些工具必须加入人工确认、权限校验和审计日志。

## 六、工具调用失败处理

常见失败：

- 参数缺失
- 参数类型错误
- 业务系统异常
- 超时
- 无权限
- 返回数据为空

模型不应该直接猜结果。工具失败时，应向用户说明失败原因，并给出下一步建议。

## 七、工具调用与工作流的关系

工具调用负责“做一个动作”，工作流负责“安排动作顺序”。

例如客服机器人：

1. 识别用户意图
2. 查询订单
3. 判断是否符合退款规则
4. 生成处理建议
5. 人工确认后提交工单

其中查询订单、提交工单是工具；整个处理过程是工作流。

## 八、本章实践任务

实现一个“订单查询助手”：

- 定义 getOrderStatus 工具
- 参数包含 orderId
- 服务端模拟查询订单状态
- 模型根据结果生成用户可读回复
- 记录工具调用输入、输出和耗时
